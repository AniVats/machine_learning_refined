Such systems include\ncloud services as well as sensor networks. Their scalability and\nreliability requirements mandate that these systems are both sharded and\nreplicated. Also, these systems evolve quickly as a result of changes in\nworkload, adding functionality, deploying new hardware, and so on. While\nthese systems are useful, they can behave in erratic ways and it is not\nclear that one can build mission- and life-critical systems this way.\n</p>\n\n<p>\nI am investigating new techniques for modular development of reliable\nand scalable systems, and call these <i>transformations</i>.  A simple\nexample of a transformation is to take a client-server system and\ntransform it into a system in which the server is replicated in some\nfashion (chain replication, Paxos, etc.).  Another kind of transformation\nthat that I worked on is turning a system that can survive crash\nfailures into one that can survive the same number of Byzantine failures.\nAnother kind of transformation would be one in which a server is sharded\nin order to increase its scalability.\n</p>\n\n<p>\nJoint work with Deniz Altinbuken and Stavros Nikolaou.\n</p>\n\n<h3>Supercloud: Combining Cloud Resources into a Single Cloud</h3>\n\n<p>\nInfrastructure as a Service (IaaS) clouds couple applications tightly\nwith the underlying infrastructures and services. This vendor lock-in\nproblem forces users to apply ad-hoc deployment strategies in order to\ntolerate cloud failures, and limits the ability of doing virtual machine\n(VM) migration and resource scaling across different clouds, and even\nwithin availability zones of the same cloud.\n</p>\n\n<p>\nThe Supercloud is a cloud service comprising resources obtained from\nseveral diverse IaaS cloud providers.  Currently, the Supercloud has been\ndeployed using resources from several major cloud providers, including\nAmazon EC2, Rackspace, HP Cloud, and some private clouds including our\nown Fractus cluster. VMs run in a virtual network and can be migrated\nseamlessly across different clouds, with different hypervisors and device\nmodels. We go well beyond simply federating clouds, as witnessed by our\nability to migrate across cloud boundaries.  We can demonstrate that,\nbeing able to deploy applications to more regions and granting more\ncontrol to end-users, the Supercloud can reduce latency, improve network\nthroughput, achieve higher availability, and reduce cost compared to\nthe underlying cloud providers.\n</p>\n\n<p>\nJoint work with Qin Jia, Zhiming Shen, Weijia Song, and Hakim Weatherspoon.\n</p>\n\n<h3>GridControl: Monitoring and Control of the Smart Power Grid</h3>\n\n<p>\nThere are pressing economic and environmental arguments for the overhaul\nof the current power grid, and its replacement with a Smart Grid that\nintegrates new kinds of green power generating systems, monitors power\nuse, and adapts consumption to match power costs and system load.\nMany promising power management ideas demand scalability of a kind\nthat seem a good fit for cloud computing, but also have requirements\n(real-time, consistency, privacy, security, etc.) that cloud computing\ndoes not currently support.\n</p>\n\n<p>\nIn cooperation with the New England ISO,\nwe are looking to develop a monitoring system that captures streaming\ndata from a large collection of so-called Phasor Measurements Units\ninstalled in the power grid, collect the data into a datacenter, and\nperform on-the-fly fitting against a model of the power grid (so-called\nstate estimation).  This way we can provide power snapshots of the current\npower grid several times a second.\n</p><p>\n\n</p><p>\nJoint work with, among others, Ken Birman, David Bindel, Z Teo, Colin Ponce, Theodoros Gkountouvas, and, at WSU, Carl Hauser, David Bakken, and David Anderson.\n</p>\n\n<h3>Configuring Distributed Computations</h3>\n<p>\nConfiguring large distributed computations is a challenging\ntask. Efficient use of a cluster requires configuration tuning\nand optimization based on careful examination of application and\nhardware properties. Considering the large number of parameters and\nimpracticality of using trial and error in a production environment,\ncluster operators tend to make these decisions based on their experience\nand rules of thumb. Such configurations can lead to underutilized and\ncostly clusters.\n</p>\n\n<p>\nWe are working on a new methodology for determining desired hardware\nand software configuration parameters for distributed computations. The\nkey insight behind this methodology is to build a response surface\nmethodology (RSM) model that captures how applications perform\nunder different hardware and software configurations. Such a model\ncan be built through iterated experiments using the real system, or,\nmore efficiently, using a simulator. The resulting RSM model can then\ngenerate recommendations for configuration parameters that are likely\nto yield the desired results even if they have not been tried either\nin simulation or in real-life. The process can be iterated to refine\nprevious predictions and achieve better results.\n</p>\n\n<p>\nWe have implemented this\nmethodology in a configuration recommendation system for MapReduce\n2.0 applications. Performance measurements show that representative\napplications achieve up to 5\u00d7 performance improvement when they use\nthe recommended configuration parameters compared to the default ones.\n</p>\n\n<p>\nJoint work Efe Gencer and Gun Sirer.\n</p>\n\n<h3>Reliable Broadcast for Geographically Dispersed Datacenters</h3>\n\n<p>\nSprinkler is a reliable high-throughput broadcast facility for\ngeographically dispersed datacenters. For scaling cloud services,\ndatacenters use caching throughout their infrastructure. Sprinkler can\nbe used to broadcast update events that invalidate cache entries. The\nnumber of recipients can scale to many thousands in such scenarios.\nThe Sprinkler infrastructure consists of two layers: one layer to\ndisseminate events among datacenters, and a second layer to disseminate\nevents among machines within a datacenter. A novel garbage collection\ninterface is introduced to save storage space and network bandwidth.\n</p>\n\n<p>\nJoint work with Haoyan Geng.\n</p>\n\n<h3>Heterogeneous Failure Models</h3>\n\n<p>\nThe robustness of distributed systems is usually phrased in terms of the\nnumber of failures of certain types that they can withstand. However,\nthese failure models are too crude to describe the different kinds of\ntrust and expectations of participants in the modern world of complex,\nintegrated systems extending across different owners, networks, and\nadministrative domains. Modern systems often exist in an environment\nof heterogeneous trust, in which different participants may have\ndifferent opinions about the trustworthiness of other nodes, and\na single participant may consider other nodes to differ in their\ntrustworthiness.\n</p>\n\n<p>\nWe explore how to construct distributed protocols that meet\nthe requirements of all participants, even in heterogeneous trust\nenvironments. The key to our approach is using lattice-based information\nflow to analyze and prove protocol properties.  Using this approach, and\nwith the help of SMT solvers, we can adapt existing protocols to work in\nthis model.  Through simulations, we can show that customizing a protocol\nto a heterogeneous trust configuration yields performance improvements\nover the conventional protocol designed for homogeneous trust.\n</p>\n\n<p>\nJoint work with Isaac Sheff and Andrew Myers&gt;\n</p>\n\n<h3>Correct-by-construction Fault-tolerant Distributed Systems</h3>\n\n<p>\nFault-tolerant distributed systems, algorithms, and protocols are\nnotoriously hard to build.\nGoing from a specification to an implementation involves\nmany subtle steps that are easy to get wrong.\nMoreover, fault-tolerant algorithms require many sources of diversity\nin order to make them survive a variety of failures.\nProgrammers, even different ones, tend to make the same mistakes when\nthey are implementing code from a specification.\n</p>\n\n<p>\nI'm developing techniques to derive distributed algorithms by stepwise\nrefinement from specification.  Each step can be formally checked.\nMoreover, one can often make different design choices as you make these\nrefinements, leading to sources of diversity that can be exploited for\nfault tolerance.\nWith the NuPrl (formal methods) group at Cornell I'm working to have\nthis process be mostly automated, and we are already able to synthesize\na variety of executable consensus protocols.\n</p>\n\n<p>Joint work with Fred Schneider, Robert Constable, and Mark Bickford.</p>\n\n<br>\n<center>\n<h2>My books</h2>\n<table>\n<tbody><tr>\n<td><a
